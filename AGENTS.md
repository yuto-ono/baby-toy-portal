# AI Agent Instructions

## 実装方針

- 既存の Svelte 5、TypeScript、SCSS の書き方に合わせ、無関係な変更は混ぜない。
- ページ固有のコードは対象ルートの近くに置き、実際に複数箇所で使うものだけ `src/lib/` へ移す。
- 将来の再利用を予想した先回りの抽象化はしない。

## 責務の分割

- `+page.svelte` と親コンポーネントは、ページ構成、データ受け渡し、主要な状態管理を担う組み立て役に保つ。
- 独立した役割、状態、副作用、アクセシビリティ要件を持つ UI は子コンポーネントへ分割する。
- 繰り返す UI や、単体テストに価値がある UI も分割を検討する。
- 短いことだけを理由に分割せず、単純なラッパーや過剰な props 転送を増やさない。

## スコープ

- 関数、定数、状態は、使う場所にできるだけ近いスコープへ置く。
- コンポーネント全体で共有しない処理は、`onMount` やイベントハンドラなど必要なブロック内へ閉じ込める。
- 単一の lifecycle、イベントハンドラ、分岐ブロックでしか使わない関数・定数は、原則としてそのブロック内に置き、ファイル上部のスコープを不用意に増やさない。
- ただし、テストしたい純粋ロジックや複数箇所で使う処理は、読みやすさを優先して適切に切り出す。

## ロジックと副作用

- 変換、判定、並べ替え、状態遷移、数値計算などは、可能な限り純粋な TypeScript 関数へ切り出す。
- 純粋関数は DOM、Svelte の状態、ブラウザ API、外部変数を直接変更しない。
- Web Audio、タイマー、イベントリスナーなどの副作用は専用モジュールかコンポーネント境界へ閉じ込め、破棄処理も実装する。
- 自明で一度しか使わない処理は、切り出すことで読みにくくなるなら近くに置く。

## テスト

- 分岐や境界値を持つ純粋ロジックには `*.spec.ts` を追加する。
- テストは実装詳細ではなく、入出力またはユーザーから見える挙動を検証する。

## CSS設計について

- 現時点では、ページ数が少なく共通化の判断材料が足りないことと、ページごとに違う雰囲気の遊びを追加する可能性があることから、ページ間の CSS はあえて積極的に共通化しない。
- `box-sizing` や `body` の基本指定など一部重複は許容し、背景、色、余白、操作感などページ体験に関わる指定は各ルートの近くに置く。
- ページ数が増えて傾向が見えてきたとき、または同じ見た目や操作性の重複が保守性を下げるようになった時点で、共通レイアウトや共通スタイルへの切り出しを再検討する。

## 完了報告前のチェック項目

- 完了報告前に、変更した `+page.svelte` や親コンポーネントが肥大化していないか、UI の責務分割、純粋ロジックの切り出し、副作用の閉じ込めが適切かを確認する。
  - 切り出すべきと判断したものはその時点で対応し、近くに置く方が読みやすい場合はその理由を完了報告に短く書く。
- 変更後は原則として `pnpm check`, `pnpm lint`, `pnpm test` を実行する。

## 全体見直し

- ユーザーから「全体見直し」と依頼された場合は、コード全体を読み、本ドキュメントの基準に照らしてレビューし、結果をユーザーに報告する。
- ユーザーの明示的な許可があるまで、コード変更、リファクタリング、Issue 起票は行わない。
  - Issue 化する前に、既存 Issue を検索し、同じ問題や近い問題がすでに起票されていないか確認する。
